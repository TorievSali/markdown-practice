# Диаграмма прецедентов: Система управления задачами

## Описание предметной области

Система управления задачами предназначена для организации работы команды.
Пользователи могут создавать проекты и задачи, назначать исполнителей,
изменять статусы задач и отслеживать выполнение работы.
Администратор управляет пользователями системы.
Система уведомлений отправляет сообщения о важных изменениях.

## Акторы

- **Пользователь** — создаёт и редактирует задачи, работает с проектами.
- **Администратор** — управляет пользователями системы.
- **Система уведомлений** — внешняя система для отправки уведомлений.

## Прецеденты

- **Вход в систему** — авторизация пользователя.
- **Создать проект** — создание нового проекта.
- **Создать задачу** — добавление новой задачи.
- **Назначить исполнителя** — прикрепление пользователя к задаче.
- **Изменить статус задачи** — обновление состояния задачи.
- **Просмотреть задачи** — просмотр списка задач.
- **Отправить уведомление** — отправка уведомлений пользователям.
- **Управление пользователями** — администрирование аккаунтов.

## Отношения

- **include** — создание задачи требует предварительного входа в систему.
- **include** — назначение исполнителя возможно после создания задачи.
- **extend** — отправка уведомления выполняется после изменения статуса задачи.

# Контрольные вопросы

## 1. Что такое диаграмма прецедентов и для чего она используется?

Диаграмма прецедентов — это UML-диаграмма,
которая показывает взаимодействие пользователей с системой.
Она используется для описания функциональных требований системы.

## 2. Чем актор отличается от прецедента?

Актор — это внешний участник взаимодействия с системой.
Прецедент — это функция системы.

## 3. Что обозначает системная граница?

Системная граница показывает,
какие элементы входят в систему,
а какие находятся вне её.

## 4. В чём разница между отношениями include и extend?

Include означает обязательное включение одного прецедента в другой.
Extend означает дополнительное расширение поведения при определённых условиях.

## 5. Почему Mermaid не подходит для построения полноценных use case диаграмм?

Mermaid не поддерживает полноценный UML-синтаксис
для use case диаграмм.

## 6. Какой синтаксис PlantUML используется для определения актора и прецедента?

Для актора:
`actor Пользователь`

Для прецедента:
`usecase "Вход в систему"`

## 7. Как в PlantUML задать отношение include?

Пример:

`CreateTask <|-- Login : <<include>>`

## 8. Какие правила именования акторов и прецедентов вы знаете?

Акторы называются существительными.
Прецеденты называются глаголами с существительными.
